## Deferred from: code review of story-1-7-silverbullet-runtime-client-and-latency-baseline (2026-04-30)

- **No fetch timeout in the runtime client** — Critical guardrail #9 explicitly defers `AbortController`-based timeouts until the latency baseline reveals tail-latency pathologies. A hung SB connection currently hangs `client.exec` / `ping` / `probe` indefinitely; the latency harness inherits the hang. Revisit when the first p95 budget breach lands or when an operator reports a stuck server. (`src/silverbullet/client.ts:139-146,200-203`)
- **`safeReadText` reads unbounded body before truncating** — `await response.text()` buffers the entire body into memory before `parseErrorBody` slices to 2048 chars, defeating the "defend the audit log against multi-MB SB error pages" intent on the read side. Cap the read with a streamed reader (`response.body.getReader()`) when memory pressure / DoS-on-error is a documented concern. (`src/silverbullet/client.ts:239-245`)
- **`searchPagesScript` requires `silversearch` plug — no startup health-check** — If the plug is absent, `silversearch.search(...)` raises `attempt to index a nil value`, surfacing as a generic `infrastructure_error` (`EBADSTATUS`) rather than a clear "missing prerequisite" message. Add a startup probe (Story 1.11 territory) or a script-level fallback that translates the missing-global into a documented `infrastructure_error` variant. (`src/silverbullet/scripts/search-pages.lua.ts`)
- **`extractCode` only handles string `code`** — `extractCode` in `src/domain/error.ts:299-304` returns `undefined` when `err.code` is numeric or symbol-typed; libuv-wrapped errors with numeric errnos drop their classification. Pre-existing in Story 1.6, surfaced by 1.7's reliance. Coerce numeric codes via `String(err.code)` when tightening AR45's error-code coverage. (`src/domain/error.ts:299-304`)
- **`OperationReport.nfr: 'NFR1' | 'NFR2'` — `'NFR2'` arm unreachable** — `measure()` always sets `nfr: 'NFR1'` because Epic 1 is read-only; the write-side NFR2 budget surfaces only when Epic 2's edit/append/create/delete handlers ship. Either narrow the type to `'NFR1'` for now or wire NFR2 measurement when Story 2.x handlers land. (`scripts/latency-baseline.ts:45,87`)
- **`RUNTIME_ERROR_CODE` lacks `assertExhaustive` helper** — Story 1.11 will switch on these for AR39 distinct messages; without an exhaustiveness helper, adding a new code (e.g. `ETIMEDOUT` if a timeout lands) silently falls through. Add when Story 1.11 wires the switch. (`src/silverbullet/client.ts:29-40`)
- **`read-page.lua` race window between `getPageMeta` and `readPage`** — Two SB calls in sequence; if the page mutates between them, `lastModified` is older than `content`. The freshness check in Story 2.3's `edit_page` will fire on the next mutation, so the corruption window is bounded. Collapse to a single multi-statement Lua script when D3's bundling optimisation lands (`architecture.md:1486-1487`). (`src/silverbullet/scripts/read-page.lua.ts:11-15`)
- **Empty Lua table → JSON `{}` vs `[]` ambiguity** — `list_pages`, `search_pages`, and `query_config_blocks` projections start with `local out = {}` and may serialise as `{}` rather than `[]` when no rows match — depending on SB's JSON encoder. The TS types declare `ReadonlyArray<...>`. Verify SB's serializer round-trips an empty Lua table to `[]` in the first end-to-end smoke (Story 1.10's `list_pages` handler test); patch the script to force-array (`if #out == 0 then return { pages = setmetatable({}, {__jsontype = "array"}) } end`) if the bug manifests. (`src/silverbullet/scripts/{list-pages,search-pages,query-config-blocks}.lua.ts`)
- **`makeFetch` test-double cursor wraps silently + `clone()` returns `this`** — Footguns in the hand-rolled fetch double: extra calls beyond `behaviours[]` length replay the last entry; `clone().text()` would double-consume the body. Tighten when Story 1.10's handler tests exercise more elaborate sequences. (`src/silverbullet/client.test.ts:568-617`)
- **No test for `fetch` throwing non-Error errno-bearing values** — The eslint-disable comment justifies a path (plain object with `code`) that is not actually exercised; only `Error` with attached `code` is covered. Add a test if the path becomes load-bearing in production. (`src/silverbullet/client.test.ts:797-807`)
- **`buildScript` payload-alphabet safety relies on `'base64'` (not `'base64url'`)** — A future refactor swapping to `Buffer.from(...).toString('base64url')` would introduce `-` / `_` characters that DO need Lua-string-literal handling. Add a runtime guard (`assert(/^[A-Za-z0-9+/=]+$/.test(payload))`) or an inline comment in the function body when the encoder is next touched. (`src/silverbullet/envelope.ts:41-42`)

## Deferred from: code review of story-1-9-freshness-state-module (2026-05-01)

- **Consumer-side handling of evicted-pre-edit case** — `get` does not bump recency (intentional per AC2 / Critical Guardrail #2), so a hot edit-target that was `read_page`'d once and then repeatedly `edit_page`'d can age out of the freshness map under load. Stories 2.3 (`edit_page`) and 2.5 (`delete_page`) must explicitly handle the `freshness.get(ref) === undefined` branch — likely treating it as a stale snapshot (fail-closed) per the freshness-invariant spirit, but the policy needs to be pinned at the consumer. Cross-reference when those stories' specs are drafted. (`src/freshness/state.ts:82-84`)
- **NaN / out-of-range `Date` validation at the consumer boundary** — `touch(ref, at)` stores any `Date` instance verbatim, including `new Date(NaN)` and `new Date(8.64e15 + 1)`. Downstream comparison `lastModified > storedAt` returns `false` for NaN (fail-open: page treated as fresh). The freshness state itself is by-design a dumb store (AC3), so the guard belongs at the touch site (Story 1.10's `read_page` handler — which gets `at` from `clock.now()`, so the risk is bounded but unpinned). Add a `Number.isFinite(meta.lastModified.getTime())` check at consumer boundaries when those handlers ship. (`src/freshness/state.ts:67-72`)
- **No `forget(ref)` after `delete_page` success** — Successful `delete_page` leaves the freshness entry behind, occupying a capacity slot until LRU-evicted. If the same page name is recreated, the stale `lastReadAt` from the prior incarnation persists and could shape the next freshness check. Spec scope-boundaries already defer this (line 470, 484); reaffirming as a Story 2.5 cross-reference rather than a new entry. (`src/freshness/state.ts:19-22`)
